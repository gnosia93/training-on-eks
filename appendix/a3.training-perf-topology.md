1. 전송 메커니즘의 근본적 차이
* write (Push 방식): 송신 측 GPU가 데이터를 준비하자마자 수신 측 GPU 메모리로 직접 쏴버리는(Push) 방식입니다. 하드웨어 가속을 최대한 활용하며 지연 시간이 가장 짧습니다.
* read (Pull 방식): 수신 측 GPU가 송신 측 GPU의 메모리에 접근하여 데이터를 직접 읽어가는(Pull) 방식입니다. 데이터가 준비되었는지 확인하는 과정과 읽어오는 과정에서 추가적인 지연 시간(Latency)이 발생하며 대역폭 효율이 급격히 떨어집니다.


#### 니트로 EC2의 태생적 한계 ####
* 하이퍼바이저의 간섭: Nitro 하이퍼바이저가 보안과 격리를 위해 GPU 간의 직접 쓰기(Direct Write) 권한을 제한합니다.
* 결과: 하드웨어는 NVLink라는 초고속 도로를 갖고 있지만, 규제(가상화 설정) 때문에 속도 제한이 걸린 read 모드로만 달려야 하는 상황입니다. NVIDIA NCCL 기술 문서에서도 가상화에 따른 이러한 성능 저하를 명시하고 있습니다.

#### 니트로 EC2에서도 write를 쓰는 법 (하지만 어려움) ####
* 호스트 커널 수준에서 IOMMU와 ACS를 비활성화해야 합니다. (AWS 인스턴스 타입에 따라 사용자가 제어 불가능한 경우가 많습니다.)
* NCCL 환경 변수를 극도로 세밀하게 튜닝해야 합니다

#### 팁 ?? ####
* 순수 성능이 제일 중요하다면 → 베어메탈(p4d.metal)로 가서 write 모드를 씁니다.
* 가성비와 운영이 중요하다면 → 니트로 EC2에서 파드를 쪼개 GDRDMA(Write 방식)를 활용해 성능 손실을 메웁니다
